# AI Novel to Script 路演 PPT 制作事实包

> 用途：把本文档连同指定图片和截图一起提供给 PPT 生成 AI，作为最终路演 PPT 的唯一事实来源。
>
> 路演时长：15 分钟。
>
> 项目阶段：作品已通过评审，进入最终路演与面试环节。
>
> 文档版本：2026-06-18。

---

## 0. 给 PPT 生成 AI 的任务指令

请根据本文档制作一份中文项目路演 PPT，要求如下：

1. 受众是项目最终评审老师，既关心产品价值，也关心技术实现、工程质量和团队判断。
2. 总路演时间为 15 分钟，建议制作 10-12 页，目标讲述时间控制在 14 分 20 秒左右。
3. 叙事顺序必须是：用户问题 → 产品方案 → 核心演示 → 技术实现 → 工程亮点 → 结果证据 → 边界和下一步。
4. 核心定位是“结构化剧本资产工作台”，不要把项目描述成普通 AI 聊天工具或单 Prompt 文本生成器。
5. 页面风格使用深色石墨灰、青色信号线和少量琥珀色警告，呈现专业影视制作中控台气质。
6. 不使用紫蓝色 AI 渐变、玻璃拟态、大面积营销卡片、夸张机器人插图或无意义数据大屏。
7. 每页只表达一个结论，页标题使用结论句，不使用空泛标题，例如不要只写“项目背景”“技术架构”。
8. 优先使用流程图、架构图、产品截图、结构化数据示例，不要堆砌代码截图和大段文字。
9. 不得编造市场数据、用户数量、准确率、效率提升比例、模型成本或性能数字。
10. 本文标记为“待团队填写”的内容必须保留占位提示，不能自动补数字。
11. 结构校验不等于文学质量评估；规则兜底不等于 AI 高质量输出；Mock 数据不能描述为真实生成。
12. Redis 已进入 Docker Compose，但当前核心业务未实际使用，不得把 Redis 写成已落地的缓存、分布式锁或任务状态方案。

建议生成内容包括：

- 每页标题。
- 每页 3-5 个核心信息点。
- 推荐图示或截图位置。
- 演讲者备注。
- 每页建议时间。
- 最终输出可编辑 PPTX。

## 1. 项目基本信息

### 1.1 项目名称

- 英文名称：AI Novel to Script
- 推荐中文名称：AI 小说转剧本工作台
- 产品界面名称：Novel2Script / Director Command Room

### 1.2 一句话定位

面向网文作者、IP 编辑和内容制作团队的 AI 改编工作台，把中长篇小说转化为可编辑、可校验、可追溯、可导出的 Scene 级结构化剧本资产。

### 1.3 不是什么

- 不是普通 AI 聊天框。
- 不是一次 Prompt 直接生成整本剧本。
- 不是替代专业编剧的自动终稿工具。
- 不是文生图或文生视频平台。
- 不是已经完成商业验证的 SaaS 产品。

### 1.4 产品目标

把小说改编前期最耗时的结构化整理工作变成连续工作流，让用户更快获得一版可继续审阅、修改和进入下游流程的剧本资产。

### 1.5 目标用户

1. 网文作者：会写小说，但不熟悉剧本场景结构，需要快速得到第一版改编资产。
2. IP 编辑和策划：需要快速梳理人物、地点、事件、冲突和可改编场次。
3. 小型短剧、漫剧或内容工作室：需要可供分镜、配音或其他生产系统继续消费的结构化中间结果。

### 1.6 项目链接

- GitHub：https://github.com/zongzuoscc/ai-novel-to-script
- 演示视频：https://www.bilibili.com/video/BV1BpEh6YEpt/

## 2. 用户问题与产品价值

### 2.1 传统小说改编的前期工作

从小说进入剧本创作前，通常需要人工完成：

- 章节阅读与摘要。
- 人物、别名、地点整理。
- 关键事件和因果顺序整理。
- 将叙事内容拆成可拍摄 Scene。
- 为每个 Scene 确定地点、时间、人物和剧情目的。
- 编写动作和对白。
- 检查人物、格式和场景完整性。
- 整理成可以交付或被其他系统消费的格式。

### 2.2 通用大模型聊天的不足

通用聊天工具可以生成“像剧本的文本”，但难以稳定满足生产要求：

- 长文本一次生成容易遗漏事件或打乱顺序。
- 人物名称、身份和对白归属可能漂移。
- 自由文本难以被程序继续处理。
- 用户很难确认某个 Scene 来源于小说哪一部分。
- 一个局部问题往往需要重新生成全部内容。
- 缺少任务进度、结构校验、错误状态和交付格式。

### 2.3 本项目提供的核心价值

| 价值 | 含义 | 项目证据 |
| --- | --- | --- |
| 结构化 | 输出章节、实体、事件、大纲和 Scene，而不是一段自由文本 | 数据库六张核心表、统一 API 契约 |
| 可追溯 | 事件、大纲和 Scene 保留 `sourceRefs` | `story_events`、`outline_scenes`、`scene_scripts` |
| 可控返工 | 支持单个 Scene 重新生成 | `/scenes/{sceneId}/regenerate` |
| 可观察 | 长任务使用 MQ，前端通过 SSE 查看阶段和结果 | RabbitMQ、项目级 SSE |
| 可校验 | 对动作、对白、角色约束和 warnings 做基础检查 | `/validate`、ValidationReport |
| 可交付 | 生成机器可读 YAML | `/export?format=yaml` |

### 2.4 推荐价值表达

> 我们不是让 AI 取代编剧，而是把小说改编前期最耗时的结构化整理，变成可观察、可控制、可返工的生产工作流。

## 3. 完整产品工作流

### 3.1 用户主链路

```text
创建项目
  ↓
粘贴或上传小说正文
  ↓
章节切分与清洗
  ↓
章节摘要
  ↓
故事资产抽取：角色 / 地点 / 事件
  ↓
场景大纲：地点 / 时间 / 人物 / 剧情目的
  ↓
Scene 剧本：动作 / 对白 / 来源 / warnings
  ↓
结构校验
  ↓
YAML 导出
```

### 3.2 系统中的结构化资产

| 资产 | 关键字段 | 作用 |
| --- | --- | --- |
| Project | projectId、title、status | 统一项目状态和入口 |
| Chapter | chapterNo、cleanText、summary | 将长文本拆成可处理单元 |
| StoryEntity | entityId、type、name、aliases、sourceRefs | 稳定人物和地点标识 |
| StoryEvent | eventId、eventOrder、summary、sourceRefs | 保持关键事件顺序和来源 |
| OutlineScene | sceneId、seqNo、slugline、purpose、characters | 将事件转为可拍摄场景规划 |
| SceneScript | action、dialogue、validationStatus、warnings | 最终 Scene 级剧本内容 |

### 3.3 核心 ID 契约

- 项目 ID：`proj_YYYYMMDD_xxx`
- Scene ID：`S001`、`S002` 递增
- 角色 ID：`C001`、`C002` 递增
- 地点 ID：`L001`、`L002` 递增
- 事件 ID：`E001`、`E002` 递增
- 空集合统一返回 `[]`，不返回 `null`

### 3.4 Scene 大纲示例

```json
{
  "sceneId": "S001",
  "seqNo": 1,
  "title": "雨夜闯店",
  "slugline": {
    "intExt": "INT",
    "locationId": "L001",
    "timeOfDay": "NIGHT"
  },
  "purpose": {
    "plot": "建立主角与旧书店的首次连接",
    "character": "展示林舟的狼狈和急迫"
  },
  "characters": ["C001", "C002"],
  "sourceRefs": ["ch1"],
  "status": "READY"
}
```

### 3.5 Scene 剧本示例

```json
{
  "sceneId": "S001",
  "seqNo": 1,
  "title": "雨夜闯店",
  "action": [
    "林舟推门而入，浑身湿透，门铃一响。"
  ],
  "dialogue": [
    {"characterId": "C001", "line": "老板，还营业吗？"},
    {"characterId": "C002", "line": "看你找什么。"}
  ],
  "sourceRefs": ["ch1"],
  "validationStatus": "PASSED",
  "warnings": []
}
```

## 4. 当前已经实现的产品能力

### 4.1 项目和正文

- 创建项目。
- 项目列表、搜索和项目详情。
- 粘贴小说正文。
- 上传 `.txt` 或 `.md` 文件。
- 替换正文。
- 追加文本或文件章节。
- 自动章节切分和清洗。
- 章节摘要生成。

### 4.2 AI 故事资产

- AI 优先抽取角色、地点和故事事件。
- 角色和地点使用稳定 ID。
- 故事事件保留事件顺序和来源章节。
- 支持全量分析。
- 支持新增章节的增量分析。
- AI 调用失败或格式异常时使用规则兜底。
- 前端区分 AI 结果、规则兜底和 Mock 回退。

### 4.3 场景和剧本

- 根据故事事件生成 Scene 场景大纲。
- 支持增量场景大纲。
- 按 `seqNo` 生成缺失 Scene。
- Scene 内容包含动作、对白、来源、校验状态和 warnings。
- 支持单 Scene 重新生成。
- 提供 Scene 流式预览。
- 3D 故事生产地图联动章节、事件、角色和 Scene。
- 移动端降级为 2D 场景时间线。

### 4.4 工作流和交付

- RabbitMQ 异步任务。
- 项目级 SSE 实时阶段事件。
- Scene 级 SSE 流式预览。
- 项目基础结构校验。
- YAML 预览、复制和下载。
- 真实项目状态和 Mock 回退状态可见。

## 6. 值得重点讲的工程设计

### 6.1 分阶段生成，而非一次性直出

原因：长篇小说一次生成容易发生事件遗漏、人物漂移和格式失控。

方案：将生成过程拆成章节、故事资产、场景大纲和 Scene 剧本，每个阶段都持久化结构化结果。

路演表达：

> 我们把大模型从一个不可控的文本黑盒，约束进一条可以逐阶段检查的生产链路。

### 6.2 RabbitMQ 解耦长任务

原因：故事分析、大纲和 Scene 生成耗时较长，不适合让浏览器同步等待 HTTP 请求。

方案：请求快速返回 jobId，任务进入 RabbitMQ，由消费者执行，前端通过 SSE 获取状态。

当前边界：

- 任务状态目前保存在进程内 `ConcurrentHashMap`。
- 默认消费者并发为 1，当前强调稳定性，但会形成全局队头阻塞。
- 服务重启后的完整任务恢复和多实例幂等仍需增强。

### 6.3 短事务避免数据库连接池耗尽

问题：如果数据库事务覆盖整个模型调用，连接会在等待 AI 时长期被占用。

当前方案：

```text
短事务读取上下文
  → 事务外调用 AI
  → 短事务保存 Scene
```

仓库提交历史中有多次针对该问题的修复：

- `d7c1f0d` 释放 Scene 生成长事务连接
- `d4874db` 防止工作流耗尽数据库连接
- `0de09a8` 隔离 Scene 生成数据库连接
- `de5ace3` 修复 Scene 查询事务等待连接泄漏

这个部分适合作为“遇到真实工程问题并逐步修复”的案例。

### 6.4 AI 优先、规则兜底

原因：模型服务可能超时、返回非法 JSON 或未配置密钥。

方案：

- 正常情况使用 AI 生成。
- JSON 响应支持去除 Markdown 包裹和提取完整 JSON 对象。
- AI 失败时生成结构完整的规则兜底结果。
- 兜底结果产生 WARNING，并在前端明确标识。

正确表达：规则兜底保证流程可继续，不保证专业创作质量。

### 6.5 `sourceRefs` 来源追溯

场景和事件保存来源章节引用，用于回答“这场戏从小说哪里来”。

当前能力是章节级引用，不能宣称已经做到逐句事实核验。下一步可增加原文片段定位和自动引用校验。

### 6.6 基础结构校验

当前检查：

- Scene 是否缺少动作描写。
- Scene 是否缺少对白。
- 对白角色是否出现在场景大纲角色列表中。
- Scene 自身是否有 warnings。

当前不检查：

- 文学性和戏剧张力。
- 跨 Scene 人物状态一致性。
- 时间线冲突。
- 关键事件覆盖率。
- 因果链完整性。

## 7. 最新流式可靠性优化

> 状态提醒：以下优化已经在当前本地工作树实现并通过构建、测试和模拟浏览器验证，但在制作最终 PPT 前应确认团队已提交、合并并部署到演示版本。

### 7.1 修复的问题

- Scene 或项目切换后旧 EventSource 主动关闭。
- 页面卸载时关闭当前流。
- 每个 chunk 校验当前 stream request，防止旧流污染新 Scene。
- 预览缓存区分 `streaming`、`completed` 和 `failed`。
- 半截失败缓存允许重新生成。
- 原文、分析、大纲或 Scene 更新后使旧预览缓存失效。
- Scene SSE 总超时至少比 AI 请求超时多 15 秒。
- 流式任务使用独立有界线程池，不再占用公共 ForkJoinPool。
- 未收到 `[DONE]` 或 finish reason 的 EOF 判定为失败。
- 非法流式 chunk 不再静默吞掉并假装成功。
- 项目级 SSE 增加事件 ID、重连间隔和 heartbeat。
- 线程池满时快速返回“当前流式生成任务较多”。

### 7.2 验证结果

- 前端 `npm run build` 通过。
- 后端 `sh mvnw test` 通过。
- 当前后端测试总数：9 个，全部通过。
- 模拟浏览器验证：S001 流式输出过程中切换 S002，S001 连接被测试服务端确认取消，没有内容串场。
- 再次返回 S001 后会发起新流，不会复用失败的半截缓存。

### 7.3 不要夸大的地方

- 尚未在最终部署环境完成真实 MySQL、RabbitMQ、模型服务的全链路压力测试。
- SSE 当前增加了心跳和事件 ID，但还没有完整 Last-Event-ID 回放和持久化事件日志。
- 流式预览仍是独立模型调用，并不是最终持久化 Scene 的同一次调用。

## 8. 产品界面与视觉素材

### 8.1 视觉方向

产品界面被设计为“导演制作中控室”，而不是通用 AI Dashboard：

- 深色石墨灰背景。
- 青色表示活动流程和选择。
- 琥珀色表示等待和 warning。
- 绿色表示完成和可交付。
- 红色表示失败或阻断。
- 面板服务于状态、行动和数据，不做装饰性卡片堆叠。

### 8.2 界面核心区域

1. 顶部 Command Bar：连接模式、当前阶段、交付状态和主要操作。
2. 左侧 Intake Console：项目、正文和文件入口。
3. 中央 Story Production Map：章节、事件、角色、地点和 Scene 关系。
4. 右侧 Scene Inspector：动作、对白、来源和流式预览。
5. 底部 Delivery Rail：流程状态、校验和 YAML 导出。

## 9. 推荐的 11 页路演结构

### 第 1 页：小说改编最先卡在结构化整理

建议时间：40 秒。

核心内容：

- 用户不是缺少一次 AI 回答，而是缺少从原文到场景资产的连续流程。
- 一句话定位。
- 目标用户：网文作者、IP 编辑、内容工作室。

视觉建议：使用 `product-hero-command-room.png`，叠加一句核心价值。

演讲提示：

> 一部长篇小说进入剧本创作前，需要先整理章节、人物、地点、事件和场次。我们的系统把这段重复但关键的前期工作变成可检查的 AI 工作流。

### 第 2 页：我们交付的是剧本资产，不是聊天回答

建议时间：50 秒。

核心内容：

- 通用聊天输出自由文本。
- 本项目输出章节、实体、事件、大纲、Scene、校验和 YAML。
- 三个关键词：结构化、可追溯、可控返工。

视觉建议：左右对比图，“聊天回答”对“剧本资产流水线”。

### 第 3 页：分阶段工作流降低长文本失控

建议时间：55 秒。

核心内容：

- 展示完整八阶段工作流。
- 说明每一步都有明确数据结构。
- 强调 `sourceRefs` 和稳定 ID。

视觉建议：横向流程图。

### 第 4 页：真实项目从小说原文进入制作工作台

建议时间：60 秒，开始产品演示。

核心内容：

- 创建或进入预置项目。
- 展示章节切分和摘要。
- 明确页面显示“真实项目”。

视觉建议：真实工作台全屏截图或现场演示。

### 第 5 页：角色、地点和事件成为稳定中间资产

建议时间：70 秒。

核心内容：

- 展示人物、地点、事件。
- 展示实体 ID、事件顺序和来源章节。
- 说明这些资产约束后续 Scene 生成。

视觉建议：角色/地点面板和事件列表局部放大。

### 第 6 页：场景大纲把故事事件转成可拍摄单元

建议时间：70 秒。

核心内容：

- 展示 3D Story Production Map。
- 选择一个 Scene。
- 展示地点、时间、人物、剧情目的和人物目的。

视觉建议：真实地图截图加少量标注。

### 第 7 页：每个 Scene 可以检查、预览和局部返工

建议时间：75 秒。

核心内容：

- 展示动作和对白。
- 展示 `sourceRefs`。
- 展示单 Scene 重生成。
- 说明流式预览的交互价值和最终结果仍以结构化 Scene 为准。

视觉建议：Scene Inspector 截图。

### 第 8 页：校验和 YAML 让结果可以继续生产

建议时间：55 秒。

核心内容：

- 展示基础校验内容。
- 展示 warning。
- 展示 YAML 导出。
- YAML 可供审阅、版本管理或下游系统消费。

视觉建议：校验报告与 YAML 示例并排。

### 第 9 页：MQ、SSE 和短事务让长任务可运行

建议时间：120 秒。

核心内容：

- Spring Boot、RabbitMQ、AI 服务和 MySQL 架构。
- HTTP 快速返回 jobId，MQ 消费任务，SSE 推送进度。
- AI 调用期间不占数据库连接。
- Scene 流使用独立有界执行器。

视觉建议：因果清晰的架构图，不要只罗列技术 Logo。

### 第 10 页：我们解决了稳定性问题，也明确当前边界

建议时间：90 秒。

左侧可讲工程修复：

- 长事务和连接池问题。
- JSON 格式清洗和规则兜底。
- Scene 流生命周期、超时、异常 EOF 和 heartbeat。
- 当前 9 个后端测试通过。

右侧主动说明边界：

- 结构校验不是文学质量评估。
- 任务状态尚未持久化。
- 默认 MQ 全局串行。
- 流式预览与正式 Scene 仍是两次模型调用。

视觉建议：已解决 / 下一步双栏。

### 第 11 页：从结构化初稿走向质量评测和协作生产

建议时间：50 秒。

下一步优先级：

1. 建立人工标注的质量评测集。
2. 减少 Scene Prompt 中的无关全项目上下文。
3. 持久化 job 状态，完善幂等和死信治理。
4. 增加人物别名、人物状态和时间线一致性检查。
5. 增加更多行业导出格式和协作审阅能力。

收尾表达：

> 我们不是让 AI 取代编剧，而是把改编前期的结构化整理，变成可观察、可控制、可返工的生产工作流。

## 11. 现场演示建议

### 11.1 固定演示路径

1. 进入预先准备的真实项目。
2. 展示章节切分和摘要。
3. 展示角色、地点和故事事件。
4. 展示场景大纲和 3D Story Production Map。
5. 选中一个代表 Scene。
6. 展示动作、对白、来源引用和 warning。
7. 必要时展示一个短 Scene 流式预览。
8. 执行结构校验。

### 11.2 演示原则

## 14. 当前已知边界与技术债

### 14.1 质量边界

- 校验主要是结构校验，不评价文学水平和戏剧张力。
- `sourceRefs` 当前主要是章节级引用，不是逐句事实证明。
- 规则兜底结果只保证结构完整和流程继续。
- 尚未建立正式人工标注的质量评测集。

### 14.2 流式边界

- 已修复旧连接、半截缓存、超时竞争和异常 EOF 等关键问题。
- 完整的 SSE 断线事件回放尚未实现。
- 流式预览和正式 Scene 仍是两次模型调用，可能产生内容差异和额外成本。

### 14.3 任务系统边界

- job 状态保存在进程内，重启后历史状态不完整。
- 项目锁是本 JVM 的 `ReentrantLock`，不支持多实例互斥。
- RabbitMQ 默认消费者并发为 1，不同项目也会互相排队。
- 尚未实现完整死信队列、任务审计和人工重放。

### 14.4 性能边界

- 每个 Scene 当前会携带全项目实体和事件，Prompt 随项目规模增长。
- Scene 批量生成存在重复上下文读取和部分 N+1 查询。
- Three.js 资源包较大，前端构建中 `three` chunk 超过 1 MB 未压缩。

### 14.5 安全与生产化边界

- 当前是实训项目，不等同于完成版权、隐私和多租户安全治理。
- 正式产品需要明确第三方模型数据传输、原文保留和删除策略。
- API Key 必须通过环境变量管理，不能进入前端、仓库、PPT 或演示录屏。

## 15. 下一步路线图

### 第一优先级：质量评测

- 建立真实小说样例集。
- 人工标注关键事件、人物、场景和一致性问题。
- 评估事件覆盖率、角色一致性、因果连贯、格式合规和人工修改量。

### 第二优先级：减少模型上下文和成本

- Scene 只读取 `sourceRefs` 对应事件。
- 只提供当前 Scene 使用的人物和地点。
- 增加必要的项目级短摘要和相邻 Scene 上下文。
- 记录输入/输出 token、首 token 和总耗时。

### 第三优先级：任务系统生产化

- MySQL 持久化 job 状态机。
- 支持服务重启后的任务查询和恢复。
- 不同项目有限并行，同一项目保持幂等。
- 增加死信队列、失败分类和人工重放。
- 多实例 SSE 事件广播和断线恢复。

### 第四优先级：剧本专业能力

- 人物别名和人物状态一致性。
- 时间线、地点和因果冲突检测。
- 关键事件覆盖和省略说明。
- DOCX、PDF、Final Draft 等导出格式。
- 多人批注、版本对比和协作审阅。

## 16. 可直接使用的开场和收尾

### 开场，约 35 秒

> 一部长篇小说进入剧本创作前，编剧往往需要先完成章节梳理、人物和地点归纳、事件排序以及场次拆分。这些工作重复，但又决定后续质量。我们做的 AI 小说转剧本工作台，面向网文作者、IP 编辑和内容团队，把小说转成可编辑、可校验、可导出的 Scene 级剧本资产。接下来我们会用一个真实项目展示从原文到 YAML 交付的完整过程。

### 收尾，约 25 秒

> 我们的目标不是让 AI 取代编剧，而是把改编前期最耗时的结构化整理，变成可观察、可控制、可返工的工作流。当前版本已经打通从原文到 Scene 级资产的完整链路，下一步会重点建立质量评测集，降低模型上下文成本，并完善人物一致性和任务持久化。
