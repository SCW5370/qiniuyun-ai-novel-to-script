一、方法论与基线界定

先把"和谁比、怎么测、为什么可信"讲清楚,再上数字。

1.1 基线:原始仓库到底优化了没有?

我们用 git 精确界定基线 —— 原始仓库(upstream/main)在三个最耗时的阶段全部是串行的:

阶段原始仓库 (upstream/main)当前 (HEAD)
章节摘要串行 for 循环逐章调用chapterSummaryExecutor 并发=4
实体/事件抽取串行 for 批次逐个调用storyAssetExecutor 并发=4
场景剧本串行逐场景调用sceneScriptExecutor 并发=4 + 背压
三个线程池配置类不存在新增 3 个 *Config(本次优化)
场景大纲串行故意保留串行(顺序上下文依赖)

验证命令:git diff --stat upstream/main...HEAD 显示三个 *Config 均为新增;git show upstream/main:.../ChapterSummaryService.java 为纯串行 for 循环。基线并非"零优化",但并发提速这一块确为本次工作。

1.2 怎么测才可信

  • 用真实线程池:基准程序用 java.util.concurrent.ThreadPoolExecutor,参数与生产 ThreadPoolTaskExecutor 一一对应(core=max=C、无界/有界队列、CallerRunsPolicy),测的是 JVM 真实调度,不是公式。
  • 把一次大模型调用建模成"固定时延 ± 15% 抖动"的任务。对这类"等待型"任务,墙钟 ≈ ⌈N/有效并发⌉ × 单次时延,随时延线性缩放 —— 所以加速比与时延无关。我们用较小时延快速实测,再按线性投影回真实秒级时延。
  • 每个测点取 3 次最优,降低机器噪声。
  • 单次时延是清晰标注的输入(摘要 4s / 抽取 8s / 大纲 10s / 场景 12s),可被真实 elapsedMs 日志一键替换重算。
为什么这套方法成立

线程池的调度行为与"任务内部做什么"无关 —— 把任务换成真实模型调用也好、换成 sleep 也好,池子的并发、排队、背压语义完全一致。隔离变量、复现行为,是性能工程的标准做法,也让数字可被任何人在本机重跑验证。

二 · 端到端结果

三、最佳并发度:压测 + 限流建模

"并发设多少"不是拍脑袋。两张图回答:本地能多快、模型让你多快。

3.1 线程池并发度扫描(N=16 个独立调用)

等待型任务在本地线程池里几乎线性扩展(贴近理想线),但效率(加速比/并发)从 C=4 的 96% 一路降到 C=12 的 73% —— 越往后每加一个线程,对模型的并发压力越大,收益越薄。

3.2 叠加模型并发上限 = 4(限流)

结论:最佳并发度 = 匹配模型的并发/限流上限

无限流时本地可一直加速,但真实模型有并发上限。一旦超过(此处建模为 4),墙钟从 C=4 的 1322ms 到 C=16 的 1188ms 几乎不动(仅 10%),多出来的并发只会撞限流(429)、被拒。所以我们选「有界并发=4」(可配 *_CONCURRENCY 环境变量)—— 既吃满收益,又不把模型打挂。这是判断,不是默认值。

4.2 背压:有界队列 + CallerRunsPolicy

24提交任务
4队列打满后 caller-runs 次数
0任务拒绝 / 丢失

故意把队列设小(=4)制造过载:队列打满时,提交线程亲自执行任务,形成自然背压 —— 既不丢任务,也不让队列无限膨胀打爆内存。生产用 queue=100 作安全阀,正常短篇不会触发,极端负载下优雅降速而非崩溃。

如何把"假设时延"换成"真实数字"

后端每阶段已打 elapsedMs 日志。跑一次真实分析,取各阶段单次耗时,作为 L_SUMMARY/L_ASSET/L_OUTLINE/L_SCENE 参数重跑即可 —— 报告里的加速比结构不变(因为它由波次数 ⌈N/C⌉ 决定),只是绝对秒数被真实值替换。方法可信,数字可校准。
